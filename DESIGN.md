# Design Document — Invoice & Payment Service

## 1. Data Model

Eight tables model a multi-tenant billing API, all scoped by `business_id`.

```
┌────────────────┐      ┌────────────────┐      ┌──────────────────┐
│   businesses   │──┬──▶│   customers    │◀─────│    invoices      │
│                │  │   └────────────────┘      │                  │
│ PK: id (uuid)  │  │                           │ PK: id (uuid)    │
│ name, ts       │  │                           │ FK: customer_id  │
└────────────────┘  │                           │ FK: business_id  │
                    │                           │ state, due_date  │
                    │                           │ total_cents (int) │
                    │                           └─────────┬────────┘
                    │                                     │
                    │   ┌────────────────┐               │ has many
                    ├──▶│   api_keys     │               ▼
                    │   │ key_prefix     │      ┌────────────────────┐
                    │   │ key_hash       │      │   line_items        │
                    │   │ revoked_at     │      │ qty, unit_amt_cents │
                    │   └────────────────┘      └────────────────────┘
                    │
                    │   ┌─────────────────────┐      ┌────────────────────┐
                    └──▶│ webhook_endpoints   │──┬──▶│ webhook_deliveries │
                        │ url, secret         │  │   │ event, payload     │
                        └─────────────────────┘  │   │ status, attempts   │
                                                 │   └────────────────────┘
                                                 │
                              ┌──────────────────────────┐
                              │   payment_attempts        │
                              │ PK: id (uuid)             │
                              │ FK: invoice_id            │
                              │ idempotency_key (unique)  │
                              │ status, card_token        │
                              │ psp_reference, failure_code│
                              │ request_body (text)       │
                              └──────────────────────────┘
```

**Primary key strategy:** UUIDs generated by Postgres `gen_random_uuid()` (requires `pgcrypto`). Avoids sequence contention, is safe across distributed writers, and does not leak row counts.

**Key indexes:**
- `api_keys.key_prefix` — O(1) auth lookup
- `invoices.business_id` + `invoices.state` — list filters
- `payment_attempts.idempotency_key` — unique; idempotency lookup
- `payment_attempts.invoice_id` — join on invoice detail
- `webhook_deliveries.webhook_endpoint_id` — retry queries

**Why int64 cents (not float/decimal):** IEEE 754 floats cannot represent all decimal values exactly. At scale, rounding compounds: $0.10 + $0.20 ≠ $0.30 in float arithmetic. Integer cents eliminate this class of bugs entirely. `NUMERIC`/`DECIMAL` is safe in SQL but adds serialization complexity at the Go layer; int64 is the industry standard (Stripe, Dodo, Square all use integer minor units).

**At 100× scale:**
- Add connection pooling (PgBouncer) and read replicas for list endpoints.
- Partition `payment_attempts` and `webhook_deliveries` by `created_at` (time-range queries dominate).
- Add a composite index on `(invoice_id, status)` for the in-flight check in the pay path.
- Move webhook delivery to a dedicated worker with a persistent queue (SQS/Redis Streams) instead of in-process goroutines.

## 2. Invoice State Machine

```
                         finalize
                   ┌─────────────────────┐
                   │                     ▼
    ┌───────┐      │              ┌───────────┐     pay (PSP ok)     ┌──────────┐
    │ draft │──────┘              │   open    │─────────────────────▶│  paid    │
    └───┬───┘                     └─────┬─────┘                      └──────────┘
        │                               │                            (terminal)
        │ void                          │ void
        ▼                               ▼
    ┌──────────┐                  ┌──────────┐
    │  void    │                  │  void    │
    └──────────┘                  └──────────┘
    (terminal)                    (terminal)

    open ──▶ uncollectible   (reserved; not implemented)
```

| From | To | Trigger | Reversible? |
|------|----|---------|-------------|
| draft | open | `POST /invoices/{id}/finalize` | No — once open, cannot go back to draft |
| draft | void | `POST /invoices/{id}/void` | No — void is terminal |
| open | paid | `POST /invoices/{id}/pay` + PSP success | No — paid is terminal |
| open | void | `POST /invoices/{id}/void` | No — void is terminal |

**Terminal states:** `paid`, `void`, `uncollectible`. No transitions out of these. Any attempt returns HTTP **409** with error code `invalid_state` or `invalid_transition` and a message indicating the current state.

**How invalid transitions are rejected:** The handler reads the current state before mutation. If the current state is not in the allowed set for that operation, it returns 409 immediately without performing any write. For the pay path, this check happens inside a `SELECT FOR UPDATE` lock, so the read is authoritative.

## 3. Payment Correctness & Failure Modes

### (a) Two clients call POST /invoices/{id}/pay simultaneously

**Outcome:** Exactly one succeeds; the other gets 409.

**Mechanism:** `SELECT * FROM invoices WHERE id = ? AND business_id = ? FOR UPDATE` inside a transaction. The first transaction acquires the row-level lock, inserts a `pending` payment_attempt, and commits. The second transaction blocks on `FOR UPDATE` until the first commits; it then sees an in-flight `pending` or `succeeded` attempt and returns 409 `invalid_state`.

**Why `SELECT FOR UPDATE`:** Deterministic serialization without retry storms. Optimistic concurrency (version column) would require the second client to retry, adding latency and complexity. Advisory locks are session-scoped and don't compose well with connection pooling.

### (b) PSP times out (tok_timeout, 30s sleep)

**What the endpoint returns:** HTTP 402 `{"status":"failed","code":"timeout"}` after ~10 seconds.

**State left behind:** `payment_attempt.status = "failed"`, `payment_attempt.failure_code = "timeout"`, invoice stays `open`.

**How the caller finds out:** The 402 response is immediate feedback. Invoice remains payable — the client can retry with a new idempotency key. A `webhook invoice.payment_failed` is also dispatched asynchronously.

**Implementation:** `context.WithTimeout(r.Context(), 10*time.Second)` wraps the PSP HTTP call. When the deadline fires, `http.Client.Do` returns `context.DeadlineExceeded`, which the handler maps to the `timeout` failure code.

### (c) PSP returns success but service crashes before persisting

**Situation:** The `payment_attempt` is in `pending` status, the PSP has charged the card, but the attempt was never updated to `succeeded` and the invoice was never set to `paid`.

**On retry (same idempotency key):** The handler finds the existing `pending` attempt in the idempotency lookup and returns HTTP 202 `{"status":"pending","message":"payment is already being processed"}`.

**Does the customer get charged twice?** Not by our service — the idempotency check prevents a second PSP call. However, the PSP has already charged. This is the "at-least-once delivery" gap.

**Production fix:** Pass our `payment_attempt.id` as the PSP's own idempotency key. Add a reconciliation job that polls `payment_attempts WHERE status = 'pending' AND created_at < now() - 5min` and queries the PSP for the final outcome.

### (d) Idempotency key reused with different request body

**What we do:** Return HTTP **422** `{"error":{"code":"idempotency_conflict","message":"idempotency key reused with different request body"}}`.

**Detection:** We store `request_body` (JSON of `{card_token}`) on the `payment_attempt`. On idempotency hit, we deserialize the stored body and compare `card_token` to the incoming value.

### (e) Invoice in `paid` state receives another POST /pay

**What happens:** The handler acquires `FOR UPDATE`, reads `state = paid`, immediately rolls back and returns HTTP **409** `{"error":{"code":"invalid_state","message":"invoice is not in open state, current state: paid"}}`. No PSP call is made.

### Concurrency mechanism summary

| Mechanism | Chosen? | Why / why not |
|-----------|---------|---------------|
| `SELECT FOR UPDATE` (row lock) | **Yes** | Deterministic, no retry needed, minimal lock window |
| Optimistic concurrency (version col) | No | Requires client retry loop; under contention retries compound |
| Advisory locks | No | Session-scoped; break with connection pooling |
| Serializable isolation | No | Broad scope; retries on any row conflict in the transaction |
| Status-conditional UPDATE | Possible | `UPDATE ... WHERE state = 'open'` + check `rows_affected`. Simpler but loses the in-flight check for pending attempts |

## 4. Webhook Design

**Events fired:**
- `invoice.created` — after invoice + line_items committed
- `invoice.paid` — after PSP success persisted
- `invoice.payment_failed` — after PSP failure or timeout persisted

**Payload shape:**
```json
{
  "event": "invoice.paid",
  "created_at": "2026-05-30T10:00:00Z",
  "data": { "invoice_id": "...", "psp_reference": "..." }
}
```

**Signing scheme:**
- Algorithm: HMAC-SHA256
- What is signed: The raw JSON payload body (byte-for-byte)
- Headers sent: `X-Webhook-Signature: sha256=<hex>`, `X-Webhook-Timestamp: <unix seconds>`
- Replay protection: Receiver should reject requests where `abs(now - timestamp) > 5 minutes`

**Retry policy:**

| Attempt | Delay after previous | Cumulative |
|---------|---------------------|------------|
| 1 | 0s (immediate) | 0s |
| 2 | 30s | 30s |
| 3 | 5 min | 5m30s |
| 4 | 30 min | 35m30s |
| 5 | 2 hours | 2h35m30s |

Total budget: 5 attempts, ~2.5 hours. After exhaustion, `webhook_delivery.status` is set to `failed`.

**What happens to exhausted webhooks:** They remain in the `webhook_deliveries` table with `status = 'failed'`. The business can reconcile by querying the invoice API (GET invoices filtered by state) or a future "replay webhook" endpoint.

**Decoupling:** `go webhook.Dispatch(...)` runs in a background goroutine. The API response is returned immediately after DB writes. The webhook delivery lifecycle is independent of request latency. In production, this would move to a queue worker for durability.

## 5. API Key Model

| Aspect | Design |
|--------|--------|
| **Generation** | 32 cryptographically random bytes → hex → prefix `sk_` (67 characters total) |
| **Storage** | `key_prefix` (first 11 chars) for lookup + `key_hash` (SHA-256 hex of full key) for verification. Raw key is **never stored**. |
| **Transmission** | Shown exactly once: at seed on first boot (logged to stdout). Subsequent requests use `Authorization: Bearer <key>` over HTTPS. |
| **Rotation** | Issue a new key for the same business, then revoke the old one (set `revoked_at`). Both keys work during the overlap window. |
| **Revocation** | Set `revoked_at` timestamp. Auth middleware filters `WHERE revoked_at IS NULL`. |
| **Blast radius** | One key → one `business_id`. Compromise exposes only that tenant's data. Cross-tenant access is impossible by design. |

## 6. What I Cut and Why

1. **Refunds / partial payments** — Requires PSP reversal, negative line items, and a new state (`refunded`). Adds significant complexity for the assignment scope; mentioned here as the first feature to add.
2. **Production rate limiting** — Token bucket or leaky bucket per API key in Redis. Omitted to keep the stack dependency-free; discussed here.
3. **Dunning / uncollectible automation** — The state exists but no cron or background job transitions invoices after N failed attempts. Would add a scheduled worker.
4. **Idempotency key expiry (TTL)** — Keys live forever. In production, expire after 24-48 hours to reclaim storage and allow legitimate retries with new bodies.
5. **Multi-currency / FX** — All amounts are USD cents. Supporting currencies requires a `currency` column, FX rates, and settlement math — explicitly out of scope.

## 7. Production Readiness Gaps

1. **Observability** — Structured logging (zerolog or slog), RED metrics (request rate/error/duration per endpoint via Prometheus), distributed tracing (OpenTelemetry spans on PSP call and webhook delivery). Currently only stdlib `log` is used.

2. **Rate limiting** — Per-API-key sliding window or token bucket, enforced in middleware or at the edge (API gateway). Prevents abuse and protects the database from thundering herds.

3. **Audit log** — Append-only `audit_events` table recording who did what and when (state transitions, key operations, payment attempts). Critical for compliance in real billing systems.
